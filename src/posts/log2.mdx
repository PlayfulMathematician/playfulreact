A few weeks ago, I made the following algorithm for computing log base 2 of a number:

```c
static inline double fast_log2(double x)
{
    uint64_t i = *(uint64_t*)&x;
    int e = ((i >> 52) & 2047) - 1023;
    uint64_t m = i & 4503599627370495ULL;
    uint32_t f = (uint32_t)(m >> 20);
    const double k = 2.3283064365386963e-10;
    return (double)e + (double)f * k + 0.043035666028;
}
```

This algorithm might seem like black magic, but it actually exploits the way floating point numbers are stored. This essay is gonna be about this algorithm and how it works as I think it is really interesting.

# Part 1: The Plan

A double has 64 bits, but what are these 64 bits? The first bit is a sign bit, if this bit is zero the number is positive. If this bit is one, the number is negative. Now the next 63 bits utilize something called scientific notation. The number is stored as $\text{sign}* \text{mantissa}*2^{\text{exponent}} $, we have already discussed the sign, but now we need to talk about the mantissa. The mantissa is stored at the last 52 bits, as a binary number. We divide the raw 52-bit value by $2^{52}$ and add one to compute the actual mantissa. The exponent is then stored in an 11 bit number before the mantissa. The raw bits have the number 1023 subtracted from them. This allows the exponent to go from -1022 to 1023. This exponent is key, because it stores an estimate for the logarithm of a number. The floating point number 12.3 has an exponent of 3 as 8 is the biggest power of 2 less than 12.3. The logarithm of 12.3 is 3.6. Getting the exponent would be enough to get a pretty good approximation of the logarithm, but we can further.

The rules of logarithms can be used to deduce that $\log_2(\text{mantissa}*2^{\text{exponent}})$ is $\text{exponent} + \log_2(\text{mantissa})$. The mantissa is between 1 and 2. So how do we compute its logarithm? ![log2(x) vs x-1](https://files.playfulmathematician.com/1403557fd14c7764cca051eb51d3a1d56a0613da.png) Looking at this image, we can see that log base 2 is a nearly perfect straight line between 1 and 2.
$$\log_2(1) = 0, \log_2(2)=1$$, so naturally a pretty good approximation of the logarithm of the mantissa is the mantissa - 1.

Our current strategy has a bit of an issue. It consistently underestimates the logarithm, this is because mantissa - 1 consistently underestimates the log of the mantissa. How do we solve this? One strategy could be to add a constant value to our output, but what value should we choose?

Let's try to compute the maximum possible error. For values between 1 and 2, at what point is $\log_2(x) - (x - 1)$ the highest? We can solve this with calculus!
The derivative of that function is $\frac{1}{x\ln(2)} - 1$. Now we must solve for x. $\frac{1}{x\ln(2)} - 1 = 0$ $\frac{1}{x\ln(2)} = 1$
$x\ln(2) = 1$
$x= \frac{1}{\ln(2)}$.

We figured out where our error is highest. Now if we add half the error we will make it so the approximation is only wrong by at most half of the error.
So now all we must do is add 0.043035666028 and we have now computed the log base 2 of a number within 0.043035666028.

# Part 2: The Program

This is the first line of the program.

```c
uint64_t i = *(uint64_t*)&x;
```

Now, this line reinterprets the float data and pretends it is a 64 bit unsigned integer. 64 bit unsigned integers allow bit manipulation which is very useful.
So what is going on in this line?

```c
&x
```

This is a pointer to x. The value of x is stored somewhere on your computer and so this figures out where it is.

```c
(uint64_t*)&x;
```

We know where x is, but we know it must tell the system that the pointer is actually pointing to a uint64_t. We cast the pointer to a double into a pointer to a uint64_t.

```c
*(uint64_t*)&x;
```

We must know what the pointer refers to. So we dereference the pointer and now the raw bits of the float are reinterpretted as an unsigned 64 bit integer.

```c
uint64_t i = *(uint64_t*)&x;
```

We then store the data in a variable called i. Now that we have the raw data time to get the exponent.

```c
int e = ((i >> 52) & 2047) - 1023;
```

This line computes the exponent from the raw data, but how?

```c
(i >> 52)
```

We shift the exponent to the end of the floating point number. So what this bit shift computes is that it takes the first 12 bits separately.

```c
((i >> 52) & 2047)
```

This 2047 value is all 1s, except for the first bit which is a zero. This leads to the first bit being set to zero, but the next 11 are unchanged.
So why set the first bit to zero, because the first bit is a signed bit, this bitwise AND gets rid of the sign.

```c
((i >> 52) & 2047) - 1023
```

Remember doubles don't store the exponent, they store the exponent + 1023, but here we subtract 1023 to get the real exponent.

```c
int e = ((i >> 52) & 2047) - 1023;
```

We now store the exponent into e.

```c
uint64_t m = i & 4503599627370495ULL;
```

This line is rather simple. It takes the last 52-bits and stores it in a variable called m.

```c
uint32_t f = (uint32_t)(m >> 20);
```

The last 20 bits and remove them. Now currently the value of f is the mantissa - 1 multiplied by $2^{32}$ floored. So now we can finally get the mantissa.

```c
const double k = 2.3283064365386963e-10;
```

This k value is $2^{-32}$. It is time for the final step.

```c
(double)e
```

We convert e to a double.

```c
(double)f * k
```

We take f and multiply it by $2^{-32}$, this value is finally the mantissa - 1.

```c
return (double)e + (double)f * k + 0.043035666028;
```

We combine the exponent, the mantissa, and our correction term. This is how we compute the log base 2.

# Conclusion

Notice how little we relied on the properties of the numbers themselves. We utilized the actual ways these things are stored. Numbers in computers aren't just abstract ideas like they are in math, but they are an actual thing that must be put somewhere.
Numbers are fundamentally information-less, computers aren't.
We create these concrete wrappers around numbers because we have to. But these wrappers affect what constructs are obvious and what are not.
They affect the speed at which computation can occur. In a sense, bit manipulation is no less janky than addition.
It makes me wonder how much math is due to the choice of wrapper. What kind of math can we discover if we change the wrapper we use?
